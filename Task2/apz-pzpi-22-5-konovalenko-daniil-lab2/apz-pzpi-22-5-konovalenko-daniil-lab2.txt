Мобільний застосунок


2.1 Мета роботи:  
Розробити мобільний застосунок для програмної системи.  



2.2 Опис інженерних рішень

Прийняті рішення:
•	Архітектура застосунку — Modular + MVVM-подібна (спрощено, з Fragment + Adapter + Model)
•	Навігація — через BottomNavigationView
•	Фрагменти (Fragments) використовуються для:
o	Список медикаментів
o	Список пристроїв
o	Умови зберігання
o	Логи
o	Налаштування
•	Дані отримуються через REST API (HTTPUrlConnection)
•	Аутентифікація через JWT токен → токен зберігається у SharedPreferences
•	Використання RecyclerView + Adapter для відображення списків
•	Пошук реалізовано локально у кожному фрагменті (по даних у пам’яті)
•	Оновлення даних — через виклик у onResume()
















2.3 Взаємодія з іншими частинами системи

•	Мобільний застосунок взаємодіє із сервером MedicationManagement через REST API
•	Сервер надає:
•	/api/medicine — медикаменти
•	/api/iotdevice — пристрої
•	/api/storagecondition — умови зберігання
•	/api/auditlog — логи
•	Аутентифікація — через /api/auth/login → отримання JWT токена
•	Всі запити йдуть з авторизацією Bearer Token

2.5 Висновок

У рамках розробки програмної системи MedicationManagement було створено повноцінний мобільний застосунок, що забезпечує зручний доступ до основних функцій системи з будь-якого Android-пристрою.
Мобільний застосунок взаємодіє з сервером системи MedicationManagement за допомогою API-інтерфейсів, що забезпечують повний доступ до всіх функцій.
У застосунку також реалізовано можливість перемикання мови інтерфейсу, що дозволяє покращити зручність користування для різних категорій користувачів.
